# 我们花了 $129 测试了 1620 组实验：LLM 技能包到底什么时候有用？

> 一项系统性实验，揭示了 AI 编程助手中"技能"功能的真实效果。

---

## 引言

如果你用过 Claude Code、Cursor 或其他 AI 编程工具，你可能注意到了一个功能叫 **"Skills"（技能包）**——它本质上是一份结构化的领域知识文档，告诉 LLM "遇到这类任务该怎么做"。

**但它真的有用吗？什么时候有用？什么时候反而有害？**

为了回答这些问题，我们设计了一个受控实验：

| 参数 | 数值 |
|------|------|
| 测试模型 | 6 个（Haiku 3.5, GPT-4o, GPT-4.1, Gemini 2.0 Pro, Sonnet 3.5, Opus 3.5） |
| 测试场景 | 30 个科学计算任务（覆盖神经科学、天文学、基因组学等 15+ 领域） |
| 实验条件 | 15 种（5 个技能完整度级别 + 5 种错误变异 + 4 种对照条件） |
| 总试验数 | **1,620 组** |
| 总花费 | **$129.05** |

每个试验的流程：给 LLM 一个科学计算任务描述 → LLM 生成 Python 代码 → 自动化测试套件评估代码的执行正确性。

---

## 什么是"技能包"？

一个完整的技能包（Skill Package）包含 4 个层级：

```
📁 skill/
├── SKILL.md          # L1: 自然语言指导（要点、陷阱、模式）
├── scripts/          # L2: 参考代码实现
│   └── main.py
├── references/       # L3: API 文档、示例代码
│   ├── api_notes.md
│   └── examples.md
└── assets/           # L4: 完整技能包
```

我们测试了从 L0（无技能）到 L4（完整技能包）的 5 个级别。

## 发现一：技能对所有模型都有帮助，但受益程度差异巨大

![RQ1 Heatmap](../figures/fig_rq1_heatmap.png)

| 模型 | 无技能 (L0) | 完整技能 (L4) | 提升幅度 |
|------|-----------|-------------|---------|
| GPT-4o | 31.1% | 49.3% | **+18.2pp** |
| GPT-4.1 | 35.0% | 51.7% | **+16.7pp** |
| Sonnet 3.5 | 34.6% | 50.7% | +16.1pp |
| Gemini 2.0 Pro | 36.6% | 50.9% | +14.4pp |
| Haiku 3.5 | 37.7% | 48.5% | +10.8pp |
| Opus 3.5 | 38.0% | 46.8% | +8.8pp |

**关键洞察**：
- **GPT-4o 获益最大**（+18.2pp）——基线最低的模型从技能中获得了最多帮助
- **Opus 获益最少**（+8.8pp）——已经很强的模型，技能的边际收益递减
- **仅 SKILL.md 文本（L1）就贡献了大部分提升**——从 L1 到 L4 的增量越来越小

**实践启示**：如果你只有精力写一样东西，写好 SKILL.md 的自然语言描述就够了。脚本和参考文档是锦上添花。

## 发现二：模型对大多数错误有强容错性，但 `wrong_import` 是致命的

![RQ2 Tolerance](../figures/fig_rq2_tolerance.png)

我们故意在技能包中注入了 5 种常见错误：

| 错误类型 | 描述 | 平均影响 |
|---------|------|---------|
| `stale_api` | 使用过时的 API 调用 | **0pp**（完全免疫） |
| `wrong_default` | 错误的默认参数值 | **-0.1pp**（几乎无影响） |
| `logic_error` | 代码逻辑错误 | **-3pp**（轻微） |
| `missing_edge_case` | 遗漏边界情况处理 | **-3pp**（轻微） |
| `wrong_import` | 导入不存在的包名 | **-15pp**（致命！） |

**为什么 `wrong_import` 如此致命？**

因为其他错误都在"逻辑层面"——模型可以用自己的知识纠正。但错误的 import 语句（比如 `from scipy.signal import butter_bandpass`，实际上不存在这个函数）会导致代码直接 crash，模型无法通过"理解代码逻辑"来避免这个问题。

**各模型表现差异巨大**：
- **GPT-4o 最脆弱**：wrong_import 导致 -40pp 的灾难性下降
- **Opus 完全免疫**：甚至在 wrong_import 条件下表现还提升了（+9pp），说明强模型能识别并绕过错误导入

**实践启示**：在发布技能包之前，**务必验证所有 import 语句**。这是唯一能造成灾难性后果的错误类型。一个简单的静态分析检查就能避免最大的风险。

## 发现三："免疫前缀"对弱模型有用，对强模型有害

![RQ3 Conditions](../figures/fig_rq3_conditions.png)

我们测试了 4 种条件：
1. **No Skill**：完全不提供技能
2. **Full Skill**：提供正确的技能包
3. **Wrong Skill**：提供其他领域的技能包（张冠李戴）
4. **Vaccinated**：在技能包前加上免疫前缀（"以下技能可能不完全准确，请批判性地参考"）

| 发现 | 详情 |
|------|------|
| 错误技能对弱模型有害 | GPT-4o 用了错误技能后下降 5pp |
| 错误技能对强模型无害 | Opus 用了错误技能后仅下降 1pp |
| 免疫前缀帮助弱模型 | GPT-4o 加上免疫前缀后从 47% 恢复到 65% |
| **免疫前缀伤害强模型** | **Opus 加上免疫前缀后从 69% 暴跌到 53%（-16pp）** |

**为什么免疫前缀会伤害强模型？**

我们的假设：免疫前缀（"请批判性地参考这个技能"）增加了模型的"怀疑模式"。弱模型本来就倾向于无脑复制技能内容，免疫前缀帮助它们建立了适度的批判性。但强模型本来就有良好的判断力，额外的怀疑指令反而让它们过度犹豫，放弃了本来正确的参考信息。

**实践启示**：
- 对弱模型（Haiku、GPT-4o-mini 等），添加"请批判性参考"的免疫前缀是有益的
- 对强模型（Opus、GPT-4.1 等），**不要添加免疫前缀**——信任模型的判断

## 对实践者的建议

### 如果你是技能/Prompt 设计者

1. **写好 SKILL.md 是性价比最高的投入**——80% 的收益来自自然语言描述
2. **验证 import 语句**——这是唯一致命的错误类型，花 30 秒检查一下能避免灾难
3. **保持技能新鲜**——过时 API 不会造成伤害（模型会自己适配），但错误的包名会
4. **根据目标模型调整策略**——弱模型需要更多引导，强模型需要更多自由

### 如果你是工具开发者

1. **优先为弱模型添加技能支持**——弱模型 + 好技能 ≈ 强模型的效果
2. **纯文本技能就足够好**——不需要复杂的脚本打包机制，一个 Markdown 文件就够了
3. **加入 import 验证**——在技能注入前做一次简单的静态检查

### 如果你是团队管理者

1. **技能库是有价值的团队资产**——尤其对于使用低成本模型的团队
2. **投入维护技能的 import 准确性**比完善其他内容更重要
3. **不要过度工程化技能系统**——简单的 Markdown 文件 + 好的组织就够了

## 实验方法论

### 任务设计

30 个科学计算任务覆盖 15+ 学科领域，每个任务要求 LLM 生成一个完整的 Python CLI 工具：

- **神经科学**：spike 数据标准化、EEG 滤波
- **天文学**：系外行星凌星检测、小行星轨道计算
- **基因组学**：基因本体分析、DNA 甲基化处理
- **地球科学**：海洋 CTD 数据处理、地震目录分析
- **更多**：音频特征提取、蛋白质结构分析、信号处理...

### 评估标准

每个场景配有自动化测试套件（5-15 个断言），包括：
- **L1 测试**：代码能否执行？输出文件是否存在？格式是否正确？
- **L2 测试**：计算结果是否正确？边界情况处理是否正确？

通过率 = 通过的断言数 / 总断言数

### 技能生成

所有技能由 Sonnet 3.5 模型通过 conversation-to-skill pipeline 生成，经过：
1. 模拟 20 轮技术对话
2. 自动提取为结构化技能包
3. skill-metric 评分验证（24/24 满分）
4. 代码可用性端到端测试

---

## 局限性

1. **场景偏向科学计算**——web 开发、系统编程等领域的结论可能不同
2. **技能由 Sonnet 生成**——人工编写的技能质量可能更高或更低
3. **仅测试单轮生成**——实际使用中 LLM 可以多轮修正
4. **30 个场景可能不够**——部分结论的统计功效有限
